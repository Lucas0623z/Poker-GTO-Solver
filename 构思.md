# 德州扑克 GTO 模拟器项目构想

## 1. 项目目标

我要开发一个**不依赖训练大模型、只基于数学建模 + 搜索算法 + 博弈论近似方法**的德州扑克（Texas Hold'em）GTO 模拟器。

这个项目的目标不是做一个“AI 聊天陪练”，而是做一个可以：

1. **计算或近似求解策略**的核心引擎；
2. **分析不同局面 EV / 策略频率 / 范围变化**；
3. **支持后续扩展为桌面端或 Web 可视化工具**；
4. 在工程上保持模块清晰，适合通过 Claude Code 的多个 agent 分工协作。

---

## 2. 明确边界

### 2.1 不做的事

为了控制复杂度，第一阶段**不做**以下内容：

* 不训练神经网络；
* 不接入大模型做策略生成；
* 不追求一开始就完全覆盖全街（Preflop / Flop / Turn / River）所有多人局；
* 不追求商业级 solver 精度；
* 不优先做 UI 美化。

### 2.2 第一阶段要做的事

第一阶段聚焦于一个**最小可行 GTO 求解器（MVP）**：

* 先支持 **Heads-Up（两人）**；
* 先从**简化局面**开始：

  * 可先做 River 子博弈；
  * 再扩展到 Turn / Flop；
  * Preflop 最后补；
* 使用**显式规则 + 手写算法**；
* 使用 **CFR / CFR+ / MCCFR** 一类经典算法；
* 支持：

  * 范围输入；
  * 公共牌输入；
  * bet sizing 抽象；
  * 输出某节点的策略频率与 EV。

---

## 3. 核心技术路线

这个项目本质上是一个**不完全信息序贯博弈求解器**。核心路线如下：

1. **游戏建模**

   * 定义玩家、手牌、公共牌、底池、下注轮、可行动作；
   * 把德州扑克局面表示成博弈树节点；
   * 明确 chance node（发牌）与 decision node（玩家行动）。

2. **状态抽象**

   * 因为完整 Hold'em 状态空间过大，必须做 abstraction；
   * 初期使用手工 abstraction：

     * 限定 bet size 集合；
     * 限定 stack-to-pot ratio；
     * 限定范围粒度；
     * 可先只做 river 已知公共牌情形。

3. **策略求解**

   * 用 CFR 系列算法迭代逼近纳什均衡；
   * 每轮遍历信息集，更新 regret 和 average strategy；
   * 输出 exploitability 近似指标或局部收敛指标。

4. **手牌评估**

   * 需要一个高性能 hand evaluator；
   * 用于 showdown 胜负判断、equity 计算、范围对比。

5. **工程化**

   * 核心引擎与 UI 解耦；
   * 算法、规则、数据结构、测试、可视化分模块实现；
   * 保证可以让多个 Claude agent 并行工作。

## 5. 多 agent 分工设计

下面的 agent 设计目标是：**低耦合、高内聚、便于 Claude Code 分工**。

---

### Agent A：架构师 / 项目总控（Architect Agent）

#### 职责

* 维护整体架构一致性；
* 负责模块边界、接口设计、目录结构；
* 决定每阶段里程碑；
* 审查其他 agent 的 PR / 代码是否违背总体设计。

#### 输入

* 本文档；
* 当前代码库结构；
* 各 agent 的进度汇报。

#### 输出

* 架构文档；
* 模块接口定义；
* 开发优先级调整建议；
* 重构计划。

#### 关注点

* 不让算法代码和 UI / CLI 混在一起；
* 不让“先能跑”变成不可维护；
* 所有数据结构命名清晰、可扩展。

#### Claude 提示方向

> 你是项目架构师。你的目标不是直接乱写功能，而是确保整个德州扑克 GTO 模拟器在模块边界、接口、依赖方向和可测试性上保持一致。任何新增代码都要先检查是否符合整体架构。

---

### Agent B：德州扑克规则与状态建模（Game Model Agent）

#### 职责

* 实现扑克牌、手牌、公共牌、底池、回合、玩家行动；
* 构建 game state；
* 生成合法动作；
* 处理 terminal state / showdown / fold / all-in 等规则。

#### 输入

* 游戏规则定义；
* 当前阶段支持的街和下注抽象；
* 架构师给出的接口规范。

#### 输出

* `Card`, `Deck`, `Hand`, `Board` 等基础类；
* `GameState`、`Action`、`PlayerState`；
* 合法行动生成器；
* 可序列化的状态表示。

#### 关注点

* 规则正确性优先；
* 状态转移必须可测试；
* 不提前把 solver 逻辑耦合进 state 层。

#### Claude 提示方向

> 你负责德州扑克的规则引擎与状态机。重点是把游戏规则、行动合法性、状态转移做对，并为 solver 提供稳定、简洁的数据接口。禁止在这一层引入求解算法细节。

---

### Agent C：手牌评估与 Equity 计算（Evaluator Agent）

#### 职责

* 实现 5/7 张牌型评估；
* 比较两手牌强弱；
* 做范围对范围 equity 计算；
* 提供 showdown utility 计算支持。

#### 输入

* 扑克牌表示方式；
* board / hole cards 数据结构；
* 需要支持的 API 形式。

#### 输出

* hand evaluator；
* equity calculator；
* benchmark 与性能测试；
* evaluator 单元测试。

#### 关注点

* 正确性和性能都重要；
* API 尽量纯函数化；
* 结果要便于 solver 重复调用。

#### Claude 提示方向

> 你负责手牌评估器和 equity 计算模块。目标是：接口简单、结果绝对正确、性能足够高，适合在 CFR 迭代中被频繁调用。

---

### Agent D：博弈树生成与抽象（Tree & Abstraction Agent）

#### 职责

* 把局面展开成决策树；
* 设计信息集（information set）表示；
* 做 bet sizing abstraction；
* 控制树规模，避免状态爆炸。

#### 输入

* 游戏状态模块；
* 当前阶段的抽象策略；
* 求解器对树结构的需求。

#### 输出

* 节点结构；
* tree builder；
* information set key 生成器；
* bet abstraction 配置。

#### 关注点

* 树结构既要对 solver 友好，也要对调试友好；
* 信息集划分必须稳定；
* 第一阶段宁可抽象粗，也不要不可运行。

#### Claude 提示方向

> 你负责博弈树和抽象层。核心目标是：在可计算性与策略表达能力之间取得平衡，优先保证树可以稳定生成、信息集定义清晰、节点可被 solver 高效访问。

---

### Agent E：CFR / GTO 求解器（Solver Agent）

#### 职责

* 实现 CFR 基础框架；
* 再扩展到 CFR+ 或 MCCFR；
* 维护 regret、strategy sum、iteration loop；
* 输出平均策略和节点 EV。

#### 输入

* 决策树；
* information set 映射；
* utility 计算接口。

#### 输出

* 可运行的 solver 核心；
* 训练/迭代入口；
* convergence 日志；
* 策略导出结果。

#### 关注点

* 先做正确，再做快；
* 先让小局面稳定收敛；
* 每个公式和更新步骤都要可解释、可测试。

#### Claude 提示方向

> 你是求解器工程师，负责 CFR 系列算法实现。重点在于把 regret update、strategy averaging、递归遍历、terminal utility 这些核心机制写正确，并能在小规模局面上验证收敛趋势。

---

### Agent F：测试与验证（Testing & Verification Agent）

#### 职责

* 为规则层、evaluator、tree、solver 编写测试；
* 构造已知答案的小型 toy game 做验证；
* 监控 regression；
* 设计 benchmark。

#### 输入

* 各模块公开接口；
* 已知规则样例；
* toy poker / Kuhn poker / Leduc poker 对照实验。

#### 输出

* 单元测试；
* 集成测试；
* 回归测试；
* 性能和精度报告。

#### 关注点

* 每个核心模块必须能单独测；
* 优先用小博弈验证 CFR 正确性；
* 先验证算法，再上复杂扑克局面。

#### Claude 提示方向

> 你负责测试与验证。你的任务不是补几个表面测试，而是建立一套能持续发现规则错误、状态转移错误、求解器公式错误和性能退化问题的测试体系。

---

### Agent G：CLI / 可视化 / 结果导出（Interface Agent）

#### 职责

* 做命令行入口；
* 负责读取配置、启动求解、导出策略；
* 提供后续 UI 或 Web API 的基础；
* 展示范围矩阵、动作频率、EV。

#### 输入

* solver 输出结果；
* 配置文件；
* 用户输入的局面描述。

#### 输出

* CLI 命令；
* JSON / CSV 导出；
* 基础结果展示；
* 日志与调试输出。

#### 关注点

* 不污染核心求解逻辑；
* 输入输出格式稳定；
* 便于之后接前端。

#### Claude 提示方向

> 你负责交互层和结果输出。你的目标是让用户能够方便地配置一个局面、运行求解并读取结果，但不得把业务逻辑和算法逻辑硬塞进 CLI 或展示层。

## 6. Agent 协作流程

推荐协作顺序：

1. **Architect Agent** 先定义模块接口和开发顺序；
2. **Game Model Agent** 搭好基础规则与状态机；
3. **Evaluator Agent** 完成牌力评估；
4. **Tree & Abstraction Agent** 建立树和信息集；
5. **Solver Agent** 接入 CFR；
6. **Testing Agent** 全程插入验证；
7. **Interface Agent** 最后封装可用入口。

### 协作规则

* 每个 agent 只能改自己负责模块，跨模块修改必须说明原因；
* 所有公共接口变更必须先更新文档；
* 先写测试，再扩功能；
* 每次提交只解决一个明确问题；
* 禁止“顺手大重构”。

## 7. 开发阶段规划

### Milestone 1：Toy Game 验证

先不要直接上完整 Hold'em，先做：

* Kuhn Poker 或 Leduc Poker；
* 验证 CFR 实现正确；
* 跑出接近已知均衡的结果；
* 建立 regret / average strategy / exploitability 的监控方法。

**目标：证明 solver 是对的。**

### Milestone 2：River 子博弈

* 固定公共牌；
* 两人范围对抗；
* 有限 bet sizing；
* 支持 check / bet / raise / fold / call；
* 输出河牌节点策略。

**目标：证明规则层 + evaluator + tree + solver 能联动。**

### Milestone 3：Turn / Flop 扩展

* 加入后续发牌 chance node；
* 增加树规模控制；
* 尝试 MCCFR 或更强抽样方法；
* 做更合理的 abstraction。

**目标：证明系统具备扩展性。**

### Milestone 4：Preflop 与完整流程

* 定义 preflop range；
* 加入位置、盲注、stack 配置；
* 支持从翻前到河牌的全流程近似求解。

**目标：形成真正可用的 MVP。**

## 8. 关键技术难点

### 8.1 状态空间爆炸

完整德州扑克的博弈树极大，不可能暴力完整求解。

**应对策略：**

* 从 toy game 开始；
* 限制 bet sizing；
* 限制可分析街道；
* 使用 abstraction；
* 后续考虑抽样 CFR。

### 8.2 信息集设计

GTO 求解不只是在“节点”上做决策，而是在**信息集**上做决策。

**应对策略：**

* 明确玩家可观察信息；
* 将隐藏手牌和历史动作编码成稳定 key；
* 从简单版本开始，避免过早做复杂 bucket。

### 8.3 性能瓶颈

热点通常在：

* tree traversal；
* evaluator；
* regret update；
* 大量 map / hash 查询。

**应对策略：**

* 优先保证正确性；
* 之后做 profiling；
* 缓存重复计算；
* 减少对象分配；
* 热点模块可换更高性能实现。

## 9. 技术选型建议

如果重点是工程开发效率，可以考虑：

* **Rust**：性能强、内存安全、适合 solver 核心；
* **C++**：性能强，但复杂度高；
* **Python**：适合原型验证，但大规模求解性能弱；
* **TypeScript**：适合配 UI / Web，但 solver 核心不一定最优；
* **Java**：工程稳定，适合你当前开发背景，如果你更熟悉 Java，可以先用 Java 做 MVP。

### 我的建议

如果你的目标是：

* 先快速做出结构正确的 solver，
* 并且你当前主要在 Cursor + Claude Code 里高频迭代，

那么可以采用：

* **核心求解器：Java 或 Rust**
* **可视化/外层工具：TypeScript 或 Java CLI**

如果你想最大化当前开发效率，**先用 Java 做第一版 MVP** 很合理，因为：

* 你上手快；
* Claude 也更容易稳定生成结构化 Java 工程；
* 测试、接口、重构都比较顺手。

## 13. 一句话总结

这不是一个“让 AI 猜怎么打牌”的项目，而是一个**用算法逼近不完全信息博弈均衡策略的工程化 solver 项目**。核心成功标准不是 UI 漂亮，而是：

* 数学逻辑正确；
* 架构清晰；
* 可以在小规模局面收敛；
* 能逐步扩展到更真实的德州扑克场景。

### 14.2 Figma MCP 配置

**方式一：远程 MCP（推荐，支持 Code → Figma）**

```bash
claude mcp add --transport http figma https://mcp.figma.com/mcp
```

- 支持 **Code to Canvas**：在 Claude Code 中说「Send this to Figma」即可把当前 UI 转为 Figma 可编辑 Frame；
- 需要 OAuth 登录：在 Claude Code 中执行 `/mcp`，选择 figma，按提示完成授权。

**方式二：本地 MCP（支持 Figma → Code）**

1. 打开 Figma 桌面应用；
2. 设置 → Preferences → 勾选 **Enable Dev Mode MCP Server**；
3. 本地服务地址：`http://127.0.0.1:3845/sse`；
4. 添加 MCP：
   ```bash
   claude mcp add --transport sse figma-dev-mode-mcp-server http://127.0.0.1:3845/sse
   ```

**建议**：两种都配置，实现双向工作流（Code↔Figma）。

### 14.3 GitHub MCP 配置

**远程（GitHub 托管，推荐）：**

```bash
claude mcp add github --transport http https://api.githubcopilot.com/mcp/ \
  --header "Authorization: Bearer YOUR_GITHUB_TOKEN"
```

- 将 `YOUR_GITHUB_TOKEN` 换成你的 [GitHub Personal Access Token](https://github.com/settings/tokens)；
- Token 需勾选 `repo` 等权限。

**验证：** `claude mcp list` 可看到已添加的 MCP。
